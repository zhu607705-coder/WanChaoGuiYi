# Ralplan 共识计划：MVP 内容扩展

## RALPLAN-DR 摘要

### 原则
1. 数据先行：先完成 JSON 数据，再做系统逻辑
2. 史料引用：每条新增内容必须有可验证的史书出处
3. 机制差异化：每位帝皇的独特机制必须通过新系统体现
4. 向后兼容：新增内容不能破坏现有 13 位帝皇和 200 个事件

### 决策驱动
1. 区域是所有系统的空间基础，优先扩展
2. 军事系统是最复杂的新模块，需要将领、兵器、地形三层设计
3. 建筑系统融入科技/政策，不单独成系统

### 可行方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 区域→军事→科技→建筑 串行 | 每步可验证，风险低 | 周期长 |
| B: 区域+科技并行，军事+建筑并行 | 两组并行，效率高 | 需要协调依赖 |
| C: 全部并行 | 最快 | 风险最高 |

**选择方案 B：** 区域+科技一组（数据密集），军事+建筑一组（逻辑密集），两组并行。

---

## 实施计划

### 第一组：区域 + 科技（数据密集）

**Step 1: 区域扩展（17 → 60+）**
- 参考《汉书·地理志》《隋书·地理志》+ 渤海小吏原文
- 新增 43-47 个区域，覆盖：东北、西北、西南、东南沿海
- 每个区域含完整数据 + historical_layer
- 文件：`regions.json`、`historical_layers.json`

**Step 2: 科技树扩展（19 → 40+）**
- 新增 21+ 项科技，覆盖更多历史时期
- 科技解锁建筑、装备
- 文件：`technologies.json`

### 第二组：军事 + 建筑（逻辑密集）

**Step 3: 军事系统深化**
- 新增将领系统：`General.cs`、`GeneralSystem.cs`
- 新增兵种升级线：3 层 × 3 条路径
- 新增地形修正：BattleResolver 读取 terrain modifier
- 新增补给线：ArmyState.supply
- 文件：`Scripts/Military/`、`units.json`

**Step 4: 建筑系统**
- 新增 Building 数据模型：`BuildingDefinition`
- 新增建筑效果：区域产出修正、防御修正
- 建筑通过科技解锁
- 文件：`buildings.json`、`Scripts/Building/`

**Step 5: 验证**
- JSON 验证 + validate_data.py
- 引用关系检查
- 新增帝皇机制与新系统集成测试

---

## ADR

**Decision:** 采用方案 B（区域+科技并行，军事+建筑并行）

**Drivers:**
- 区域和科技是纯数据，可以独立编写
- 军事和建筑需要代码逻辑，依赖区域数据
- 两组并行可缩短周期

**Alternatives:**
- 方案 A 串行：太慢
- 方案 C 全并行：风险太高，军事系统需要区域数据才能测试

**Consequences:**
- 需要先完成区域数据才能测试军事系统
- 建筑系统的区域槽位依赖区域数据结构
