# 《万朝归一：九州帝业》项目规则

## 项目定位

- 项目暂定名：`万朝归一：九州帝业`
- 第一阶段目标：10 到 12 周做出可运行国内版 Demo。
- Demo 目标：验证“帝皇机制 + 九州地图统一 + 王朝治理 + 继承风险”的核心闭环。
- 全球版策略：只在数据结构、世界观字段、机制标签中预留扩展，不进入 MVP 开发范围。

## 开发边界

### MVP 必做

- 史域九州融合节点地图，目标 40 到 60 个区域。
- 8 位核心帝皇：秦始皇、刘邦、汉武帝、曹操、李世民、赵匡胤、朱元璋、康熙。
- 基础资源：粮食、金钱、人口、兵力、合法性。
- 基础风险：土地兼并、民变、财政崩溃、继承危机、地方割据。
- 基础系统：战争、内政、继承、法统、人才、事件、AI、胜利条件。

### MVP 不做

- 全球地图。
- 全部中国皇帝或世界帝王。
- 多人联机。
- 复杂实时战斗。
- 大型 3D 美术。
- 复杂外交、贸易、宗教和完整历史年表。

## 核心设计原则

- 每个帝皇必须有独特机制，不能只做数值差异。
- 每张地图必须有时代差异，地图数据要能表达制度、人口、地形和法统差异。
- 每次扩张必须带来治理代价，新占领地区不能立刻完整贡献税收和兵源。
- 统一目标定义为建立可延续王朝秩序，避免把占满地图当成唯一终点。

## 技术路线

- 引擎：Unity。
- 语言：C#。
- 玩法形态：2D 节点地图策略模拟。
- 数据：MVP 使用 JSON，后续可迁移到 SQLite。
- UI：Unity UI Toolkit 或 UGUI，实际选择在 Week 1 定稿。
- 存档：JSON 存档。
- 战斗：MVP 使用自动结算，暂不做战术战场。

### 文件命名

- C# 类型使用 `PascalCase.cs`，类型名与文件名一致。
- JSON 表使用 `snake_case.json` 或领域名复数形式，例如 `emperors.json`。
- 文档使用小写短横线命名，例如 `mvp-design.md`。
- 数据 `id` 使用 `snake_case`，保持稳定，避免后续存档失效。

### 数据规则

- 帝皇数据必须包含全球扩展保留字段：`civilization`、`versionScope`、`legitimacyTypes`、`mapScope`、`globalMechanicTag`。
- 地区数据必须包含土地结构、法统记忆、地方势力、民变风险和相邻区域。
- 新字段要先写入 `docs/data-contract.md`，再进入 JSON 数据表。
- 任何机制字段必须能被 UI 解释，不能只存在于策划文字。

## 验证规则

- 修改 JSON 后必须做解析验证。
- 修改 JSON 引用关系后必须运行 `python3 tools/validate_data.py`。
- 修改核心逻辑后必须运行对应测试、Unity PlayMode 测试或最小手动验收。
- 新机制只有在至少运行一次后才算完成。
- 一次验证只能形成假说；需要重复验证或独立证据后才能沉淀为通用规则。
